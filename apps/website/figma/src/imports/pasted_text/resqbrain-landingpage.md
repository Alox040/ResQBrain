  Landingpage Blueprint: ResQBrain                                                                                                                                                                                                                                                         
                                                                                                                                                                                                                                                                                             Basis-Analyse der aktuellen Seite:                                                                                                                                                                                                                                                       
  - 8 Sections, davon 2 faktisch identisch (idea + projectGoal → beide: "geordnete Inhalte, Nachvollziehbarkeit, Praxisnähe")                                                                                                                                                              
  - Status-Section zeigt nur 3 Badge-Labels — kein echter Inhalt
  - Hero-CTA "Mitwirken" erscheint bevor das Problem etabliert ist
  - Kein Trust-Signal, keine DBRD-Referenz, keine konkreten Use Cases
  - Audience-Section kommt nach Status — Reihenfolge baut kein Vertrauen auf

  ---
  Neue Section-Struktur

  ---
  Section 1 — HERO

  Ziel: Erstkontakt herstellen. Wer bist du, was machst du, warum jetzt.

  Inhalt:
  - Badge: MVP-Phase · Aktive Entwicklung
  - Headline: ResQBrain strukturiert medizinisches Einsatzwissen für Rettungsdienstorganisationen.
  - Subline: Maximal 2 Sätze. Kein Floskel-Satz. Konkret: offline-first, versioniert, organisationsspezifisch.
  - Zwei CTAs:
    - Primär: Mitwirken → /mitwirken
    - Sekundär: GitHub → externer Link
  - Kein Inline-Status-Card (aktuelle Seite überfrachtet den Hero)

  CTA-Logik: Primärer CTA ist niedrigschwellig (Umfrage, nicht Registrierung). Sekundärer CTA für technisch Interessierte.

  ---
  Section 2 — REALITY BLOCK

  Ziel: Problem konkret benennen. Nicht abstrakt — EMS-spezifisch. Resonanz bei der Zielgruppe erzeugen.

  Inhalt:
  - Titel: Warum bestehende Lösungen nicht reichen
  - Keine generischen Karten. Stattdessen: 3 konkrete Szenarien als kurze Aussagen:
    - "PDF-Protokoll, Seite 23 — im Einsatz nicht aufzufinden."
    - "Neue SOP freigegeben. Team B weiß es noch nicht."
    - "Nachbereitung passiert, wenn jemand Zeit hat."
  - Darunter: 1 kurzer Satz als Zusammenfassung — kein weiteres Cards-Grid

  CTA-Logik: Kein CTA. Diese Section soll nur Resonanz erzeugen.

  ---
  Section 3 — CORE VALUE

  Ziel: Klar und ohne Buzzwords erklären, was ResQBrain konkret anders macht.

  Inhalt:
  - Titel: Was ResQBrain bereitstellt
  - 3 Kernaussagen — funktional, nicht aspirativ:

  ┌──────────────────────┬─────────────────────────────────────────────────┐
  │     Eigenschaft      │                     Konkret                     │
  ├──────────────────────┼─────────────────────────────────────────────────┤
  │ Offline-first        │ Inhalte auf dem Gerät, kein Netz erforderlich   │
  ├──────────────────────┼─────────────────────────────────────────────────┤
  │ Versionierte Inhalte │ Jeder sieht denselben, freigegebenen Stand      │
  ├──────────────────────┼─────────────────────────────────────────────────┤
  │ Organisationskontext │ Inhalte gehören zur Organisation, nicht zur App │
  └──────────────────────┴─────────────────────────────────────────────────┘

  - Explizite Abgrenzung: ResQBrain trifft keine medizinischen Entscheidungen. Es strukturiert freigegebene Inhalte.

  CTA-Logik: Kein CTA. Nur Klarheit schaffen.

  ---
  Section 4 — LIVE MVP STATUS

  Ziel: Ehrlich kommunizieren was existiert, was nicht. Kein Overpromising.

  Inhalt:
  - Titel: Aktueller Stand — Stand April 2026
  - Explizites Status-Board (kein vages Badge):

  ✓ Lookup-Datenmodell (Medikamente, Algorithmen)
  ✓ Offline-fähige Mobile App (Prototyp)
  ✓ Bundle-System mit Versionsvergleich
  ◎ Approval-Workflow (in Entwicklung)
  ◎ Organisationsverwaltung (in Entwicklung)
  ✗ Produktionsreifer Editor (nicht gestartet)
  ✗ Mehrorganisations-Betrieb (nicht gestartet)

  - 1 Satz: Aktive Entwicklung. Kein Produktionsrelease.
  - Link: Roadmap auf GitHub →

  CTA-Logik: Kein Kaufdruck. Transparenz als Vertrauenssignal. GitHub-Link für wer mehr Details will.

  ---
  Section 5 — USE CASES

  Ziel: Zeigen wer dieses System wie nutzt. Konkrete Rollen, keine generischen User-Personas.

  Inhalt:
  - Titel: Für wen und wie
  - Zwei klare Use Cases, nicht drei generische Audience-Karten:

  Use Case A — Organisation
  - Ärztlicher Leiter gibt Adrenalin-Dosierungsänderung frei
  - System verteilt neuen Bundle-Stand an alle Geräte der Organisation
  - Einsatzkraft sieht beim nächsten App-Start: "Inhalt aktualisiert"

  Use Case B — Einzeleinsatz
  - Einsatzkraft sucht offline nach "Anaphylaxie"
  - Findet freigegebenen Algorithmus der eigenen Organisation
  - Kein Netz nötig, kein Login während des Einsatzes

  CTA-Logik: Am Ende beider Use Cases: kleiner Link Mitmachen wenn das dein Problem ist → → /mitwirken

  ---
  Section 6 — MITWIRKUNG (Primärer CTA)

  Ziel: Zum Handeln auffordern. Niedrigschwellig, ehrlich über Aufwand.

  Inhalt:
  - Titel: Mach die App besser
  - Subtext: Praxisrückmeldungen aus dem Rettungsdienst entscheiden über die nächsten Schritte.
  - Aktive Umfrage: UI & UX Feedback — April 2026 — 2 Minuten
  - Primärer CTA: Zur Umfrage → (externer Link, aktuell: forms.office.com)
  - Sekundärer Weg: Direkter Kontakt → /kontakt
  - Optionale weitere Wege (kein separates Grid):
    - GitHub-Issue öffnen
    - Fachliche Rückmeldung per Mail

  CTA-Logik: Ein primärer Button, deutlich. Kein Drei-Karten-Grid mit drei CTAs auf gleicher Ebene.

  ---
  Section 7 — TRUST LAYER

  Ziel: Glaubwürdigkeit durch Kontext, nicht durch Marketing-Aussagen.

  Inhalt:
  - Titel: Kontext und Einordnung
  - Keine Testimonials (zu früh, nicht vorhanden)
  - Stattdessen faktische Ankerpunkte:
    - Inhalte basieren auf DBRD-Algorithmen 2026 (Seed-Daten)
    - Offenes Entwicklungsmodell — Code auf GitHub einsehbar
    - Kein kommerzielles Produkt — kein Abo, kein Vendor Lock-in
    - Datenschutz: kein Patientenbezug, keine Nutzerdaten ohne Einwilligung
  - Optional: Zitat aus Feedback (sobald vorhanden) — Platzhalter vorbereiten

  CTA-Logik: Kein CTA. Vertrauen entsteht nicht durch einen Button.

  ---
  Section 8 — FAQ

  Ziel: Häufige Einwände ausräumen, bevor sie per Mail kommen.

  Inhalt (4 Fragen — priorisiert nach Reibungspunkten):

  ┌─────────────────────────────────────────┬───────────────────────────────────────────────────────────────────┐
  │                  Frage                  │                       Warum sie wichtig ist                       │
  ├─────────────────────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ Ersetzt das klinische Entscheidungen?   │ Rechtlicher und ethischer Einwand — muss explizit verneint werden │
  ├─────────────────────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ Funktioniert es ohne Internet?          │ Kernversprechen — muss früh bestätigt werden                      │
  ├─────────────────────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ Wer ist für die Inhalte verantwortlich? │ Haftungsfrage — die Organisation, nicht ResQBrain                 │
  ├─────────────────────────────────────────┼───────────────────────────────────────────────────────────────────┤
  │ Wie komme ich rein / kann ich testen?   │ Conversion-Frage — Weg zur Mitwirkung                             │
  └─────────────────────────────────────────┴───────────────────────────────────────────────────────────────────┘

  CTA-Logik: Letztes FAQ-Element endet mit Link auf /mitwirken.

  ---
  Vergleich: Vorher / Nachher

  ┌──────────────────────────┬───────────────────────────┬───────────────────────────────┐
  │          Vorher          │          Nachher          │           Änderung            │
  ├──────────────────────────┼───────────────────────────┼───────────────────────────────┤
  │ idea + projectGoal (2×)  │ core-value (1×)           │ Merged, kein Duplikat         │
  ├──────────────────────────┼───────────────────────────┼───────────────────────────────┤
  │ status (3 Badges)        │ mvp-status (Status-Board) │ Ehrlicher, konkreter          │
  ├──────────────────────────┼───────────────────────────┼───────────────────────────────┤
  │ audience (3 Personas)    │ use-cases (2 Szenarien)   │ Handlungskontext statt Labels │
  ├──────────────────────────┼───────────────────────────┼───────────────────────────────┤
  │ kein Trust-Block         │ trust-layer               │ Vertrauen durch Fakten        │
  ├──────────────────────────┼───────────────────────────┼───────────────────────────────┤
  │ Hero mit Mitwirkung-Card │ Hero ohne Ablenkung       │ Klare erste Botschaft         │
  ├──────────────────────────┼───────────────────────────┼───────────────────────────────┤
  │ CTA zu früh im Hero-Flow │ CTA nach Use Cases        │ Problem erst, Handlung danach │
  └──────────────────────────┴───────────────────────────┴───────────────────────────────┘

  ---
  CTA-Hierarchie der gesamten Seite

  Section 1 (Hero)         → Mitwirken [Primär] + GitHub [Sekundär]
  Sections 2–4             → kein CTA (Aufmerksamkeit aufbauen)
  Section 5 (Use Cases)    → weicher Link [Tertiär]
  Section 6 (Mitwirkung)  → Zur Umfrage [Primär, Wiederholung]
  Section 7 (Trust)        → kein CTA
  Section 8 (FAQ)          → Link /mitwirken [Tertiär, letzter Touchpoint]

  Regel: Primäre CTA erscheint nur 2× auf der Seite (Hero + Mitwirkung-Section). Kein Button auf jeder Section.

  ---
  Betroffene Datei für Umsetzung

  Die gesamte Inhaltsdefinition liegt in einer einzigen Datei:

  apps/website/lib/site/content.ts

  Strukturänderung bedeutet: neues Schema in content.ts, entsprechende Sections in app/page.tsx. Keine neuen Komponenten nötig — bestehende ContentCard, SectionFrame, ButtonLink sind ausreichend.
